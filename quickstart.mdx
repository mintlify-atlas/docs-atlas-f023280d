This guide walks you through everything from installation to a complete error-handling pipeline using `Result`. By the end, you will have replaced a `try/catch` block with a composable, type-safe alternative that the TypeScript compiler can verify at every step.

<Steps>
  <Step title="Install the package">
    Add `@deessejs/fp` to your project with your preferred package manager.

    ```bash
    npm install @deessejs/fp
    ```

    The package ships as both ESM and CommonJS, so it works in Node.js, Next.js, Vite, and any other TypeScript environment without configuration.
  </Step>

  <Step title="Parse JSON safely with Result">
    The `Result` type represents an operation that either succeeds (`Ok`) or fails (`Err`). Use `ok()` to wrap a success value and `err()` to wrap an error.

    ```typescript
    import { ok, err, Result } from '@deessejs/fp'

    class ParseError extends Error {
      constructor(message: string) {
        super(message)
        this.name = 'ParseError'
      }
    }

    function parseConfig(raw: string): Result<Record<string, unknown>, ParseError> {
      try {
        return ok(JSON.parse(raw))
      } catch (e) {
        return err(new ParseError(String(e)))
      }
    }

    const result = parseConfig('{"port": 3000}')
    ```

    The return type `Result<Record<string, unknown>, ParseError>` tells every caller that this function can fail — no `try/catch` required at the call site.
  </Step>

  <Step title="Check the result with isOk and isErr">
    Use the `isOk` and `isErr` type guards to narrow the result. TypeScript knows the exact shape inside each branch.

    ```typescript
    import { ok, err, isOk, isErr, Result } from '@deessejs/fp'

    class ParseError extends Error {
      constructor(message: string) {
        super(message)
        this.name = 'ParseError'
      }
    }

    function parseConfig(raw: string): Result<Record<string, unknown>, ParseError> {
      try {
        return ok(JSON.parse(raw))
      } catch (e) {
        return err(new ParseError(String(e)))
      }
    }

    const result = parseConfig('{"port": 3000}')

    if (isOk(result)) {
      // TypeScript knows: result.value is Record<string, unknown>
      console.log('Parsed config:', result.value)
    }

    if (isErr(result)) {
      // TypeScript knows: result.error is ParseError
      console.error('Parse failed:', result.error.message)
    }
    ```
  </Step>

  <Step title="Chain operations with flatMapResult">
    Use `flatMapResult` to chain operations that can each return a `Result`. If any step returns `Err`, the chain short-circuits and the error propagates automatically.

    ```typescript
    import { ok, err, flatMapResult, mapResult, Result } from '@deessejs/fp'

    class ParseError extends Error {
      constructor(message: string) { super(message); this.name = 'ParseError' }
    }
    class ValidationError extends Error {
      constructor(public field: string) { super(`Missing field: ${field}`); this.name = 'ValidationError' }
    }

    interface Config { port: number; host: string }

    function parseConfig(raw: string): Result<Record<string, unknown>, ParseError> {
      try {
        return ok(JSON.parse(raw))
      } catch (e) {
        return err(new ParseError(String(e)))
      }
    }

    function validateConfig(
      data: Record<string, unknown>
    ): Result<Config, ValidationError> {
      if (typeof data.port !== 'number') return err(new ValidationError('port'))
      if (typeof data.host !== 'string') return err(new ValidationError('host'))
      return ok({ port: data.port as number, host: data.host as string })
    }

    // Chain parse → validate in one pipeline
    // Error type is automatically widened to ParseError | ValidationError
    const config: Result<Config, ParseError | ValidationError> = flatMapResult(
      parseConfig('{"port": 3000, "host": "localhost"}'),
      validateConfig
    )

    // mapResult transforms only the success value
    const summary = mapResult(config, c => `${c.host}:${c.port}`)
    ```

    <Tip>
      TypeScript automatically widens the error type union as you chain operations. A `Result<Config, ParseError>` chained into a function returning `Result<Config, ValidationError>` produces `Result<Config, ParseError | ValidationError>`.
    </Tip>
  </Step>

  <Step title="Handle all cases with matchResult">
    `matchResult` provides exhaustive pattern matching — you must handle both the success and error branches, and TypeScript enforces it at compile time.

    ```typescript
    import { ok, err, flatMapResult, matchResult, Result } from '@deessejs/fp'

    class ParseError extends Error {
      constructor(message: string) { super(message); this.name = 'ParseError' }
    }
    class ValidationError extends Error {
      constructor(public field: string) { super(`Missing field: ${field}`); this.name = 'ValidationError' }
    }

    interface Config { port: number; host: string }

    // ... parseConfig and validateConfig from the previous step

    const config: Result<Config, ParseError | ValidationError> = flatMapResult(
      parseConfig('{"port": 3000, "host": "localhost"}'),
      validateConfig
    )

    const message = matchResult(
      config,
      // Ok branch: value is Config
      (c) => `Server ready at ${c.host}:${c.port}`,
      // Err branch: error is ParseError | ValidationError
      (e) => `Config error: ${e.message}`
    )

    console.log(message)
    // → "Server ready at localhost:3000"
    ```

    Because the error type is a discriminated union, TypeScript verifies that you have handled every variant inside the `Err` branch. Removing a `case` is a compile error.
  </Step>
</Steps>
